✓ resolvido falha na chamada ao LLM (rede, rate limit, chave inválida/ausente): config.py::get_llm() já tenta um fallback em cadeia — Gemini (oficial, gratuito, testes/prototipagem) → Anthropic → OpenAI, cada camada só ativa se a respectiva chave estiver no .env. Só se todos os provedores configurados falharem é que o nó captura a falha e relança FalhaLLMError; api.py traduz isso em HTTP 503 com "Serviço de IA indisponível, recarregue a página." — sem retry automático em loop, a mensagem persiste até o operador recarregar mais tarde. Como o checkpointer só grava progresso após um nó terminar com sucesso, o thread_id fica naturalmente pausado no último ponto bem-sucedido — recarregar e repetir a ação resume dali, sem perder nada.

Resumo dos pontos em aberto (0, + 8 resolvidos)

  1. Nenhum ponto em aberto no momento — todos os pontos identificados neste mapa já têm comportamento decidido (ver histórico de resoluções abaixo). O que falta é implementar (nodes.py/graph.py/api.py seguem stub), não decidir.

✓ resolvido "validação da entrada da própria ferramenta" — batch_id deixou de ser argumento do LLM: vem injetado do estado (nc_input.batch_id) via Annotated[AgentState, InjectedState], travando a consulta no lote sob investigação. data_inicio/data_fim validadas com datetime.fromisoformat(...) — formato inválido ou janela invertida devolve erro claro ao LLM, não mais uma busca silenciosa sem resultado.

✓ resolvido "múltiplos parâmetros fora da faixa, qual vira a 1ª pergunta" — Fase 1 (Ishikawa) mapeia contexto por categoria antes de qualquer "por quê".

✓ resolvido "CLI sem lote elegível" — vira lista web sempre visível, sem caminho de erro a tratar.

✓ resolvido "input() bloqueante" — vira interrupt() + checkpointer (SqliteSaver) + Command(resume=...), compatível com uma API web.

✓ resolvido "formato/nome do relatório" — JSON + HTML (Jinja2), reports/{batch_id}_{ts}.{json,html}, gerado só na aprovação; "pedir ajuste" preserva o ciclo anterior em ciclos_anteriores.

✓ resolvido "resposta vazia/evasiva não validada" — Camada 1 determinística (validar_resposta_operador) rejeita vazio/frases evasivas fixas antes de a resposta virar uma tentativa, com alerta "Este tipo de resposta não é aceito".

✓ resolvido "resposta vaga/não informativa não tratada" — Camada 2 agêntica (novo nó avaliar_informatividade) julga informatividade, dá até 2 tentativas por pergunta, sinaliza o operador e segue registrando as 2 tentativas se a 2ª também falhar.

✓ resolvido "falha na chamada ao LLM sem tratamento" — FalhaLLMError por nó + HTTP 503 na API + checkpoint naturalmente pausado pra retry, sem lógica extra de "salvar progresso".

Nota: os testes (test_graph.py via harness de main.py, Command(resume=...) com 11 respostas fixas) têm mecanismo decidido — o que falta é escrever o código, não decidir o comportamento (não é mais um gap de design, ver tarefa #48).
